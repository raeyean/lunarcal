# LunarCal Code & Security Audit — 2026-05-25

## Executive Summary

35 npm vulnerabilities (6 high, 29 moderate) in mobile deps, plus 3 confirmed logic bugs in production code, 2 privacy/security issues, and several robustness gaps. All high/medium severity items have concrete fixes. No remote code execution or authentication bypass vectors found — the API is public/read-only and the attack surface is mainly local device storage and input validation.

---

## A — Security

### A1 · PII in console.warn · MEDIUM
**File:** `src/utils/profileStorage.ts:75`
```ts
console.warn('[profileStorage] dropped malformed saved date', item);
```
`item` is the raw JSON object from AsyncStorage. If a crash reporter (e.g. Sentry) captures `console.warn` output in production, the user's saved date label and solar date are transmitted off-device. The label could contain a person's name or relationship.

**Fix:** Log `item.id ?? '(no id)'` instead of the full object.

---

### A2 · Math.random UUID · LOW
**File:** `src/utils/uuid.ts:7`
```ts
const r = (Math.random() * 16) | 0;
```
`Math.random()` is not cryptographically random. IDs are currently used only for local AsyncStorage record keys (not session tokens or nonces), so the practical exploit surface is minimal. However the comment calls it "RFC 4122–shaped v4" — RFC 4122 v4 requires cryptographic random. React Native exposes `crypto.getRandomValues` via the Hermes engine.

**Fix:** Replace with `crypto.getRandomValues(new Uint8Array(16))` approach.

---

### A3 · Unencrypted AsyncStorage for birth profile · LOW (architectural)
**File:** `src/utils/profileStorage.ts`, key `@lunarcal/birthProfile`

AsyncStorage is plaintext SQLite on Android and plaintext NSDictionary on iOS. Birth date, time, and gender are stored unencrypted. On rooted Android or jailbroken iOS, any app with root/file-system access can read it. The file's PRIVACY comment correctly documents the constraint but provides no mitigation.

**Recommendation (not in fix plan — requires dependency change):** Migrate to `expo-secure-store` (uses iOS Keychain / Android EncryptedSharedPreferences). Acceptable workaround: document risk explicitly in app privacy disclosure.

---

### A4 · ErrorBoundary exposes error.message to UI · LOW
**File:** `src/components/ErrorBoundary.tsx:42`
```tsx
<Text>{this.state.error?.message}</Text>
```
In production, internal error messages may expose implementation details. Limited attack surface in a React Native app, but users also shouldn't see raw JS error strings.

**Fix:** Wrap in `__DEV__` check or provide a generic fallback message.

---

### A5 · API CORS wildcard · INFO (intentional)
**File:** `apps/api/src/index.ts:13`
```ts
cors({ origin: '*' })
```
Intentional for a public read-only API. If auth is ever added, this must be tightened. Document as a prerequisite for any future auth work.

---

### A6 · x-forwarded-proto trusted · INFO
**File:** `apps/api/src/index.ts:36`
On Firebase/GCP, the load balancer strips client-supplied `X-Forwarded-Proto`, so this is safe in production. In the local emulator it could be spoofed. No action needed.

---

### A7 · No rate limiting on public API · INFO (architectural)
The API has no request rate limiting. Firebase enforces project-level quotas but an actor could exhaust free-tier invocations. Fix requires an API Gateway or middleware layer; out of scope for this plan.

## B — Correctness Bugs

### B1 · getMonthDays pads with wrong day data · HIGH
**File:** `packages/shared/src/lunar/index.ts:308,317`
```ts
for (let i = 0; i < firstWeekDay; i++) {
  currentWeek.push(getDayData(year, month, 1, month)); // ← always day 1
}
// ...trailing cells also use day 1
while (currentWeek.length < 7) {
  currentWeek.push(getDayData(year, month, 1, month)); // ← always day 1
}
```
Leading calendar cells (days before the 1st) and trailing cells (after month end) all show the lunar data for day 1 of the current month instead of actual dates from the adjacent months. The calendar header says e.g. "廿一" but the cells before April 1 all display 正月初一's data — wrong lunar dates, wrong ganzhi, wrong yi/ji.

**Fix:** Compute the correct previous/next month dates for padding cells.

---

### B2 · computeBazi doesn't validate calendar day validity · MEDIUM
**File:** `packages/shared/src/bazi/compute.ts:61`
```ts
if (y < 1900 || y > 2100 || m < 1 || m > 12 || d < 1 || d > 31) {
  throw new BaziError('INVALID_DATE', ...);
}
```
This accepts `2026-02-30` (Feb 30 doesn't exist). The API's `isValidSolarDate` correctly uses a `new Date()` round-trip check, but `computeBazi` does not. When an invalid calendar date is passed, `Solar.fromYmdHms` may throw (caught as `ENGINE_FAILURE`) or silently normalize (date overflow). Either way the error code is wrong.

**Fix:** Add `new Date()` round-trip check matching `isValidSolarDate` pattern.

---

### B3 · Notification settings hour/minute not range-validated · MEDIUM
**File:** `src/utils/notificationSettings.ts:18-22`
```ts
const parsed = JSON.parse(raw);
return { ...DEFAULT_SETTINGS, ...parsed };
```
Spread accepts any JSON shape from AsyncStorage. Corrupted or manually edited storage could set `hour: 99` or `minute: 75`. `expo-notifications` would receive an invalid `Date` trigger and silently fail to schedule — no error surfaced to the user.

**Fix:** Validate `hour` 0–23 and `minute` 0–59 after parsing; fall back to defaults on out-of-range values.

---

### B4 · Deity route accepts day 30 for 29-day lunar months · LOW
**File:** `apps/api/src/routes/deity.ts:60`
```ts
if (lunarMonth < 1 || lunarMonth > 12 || lunarDay < 1 || lunarDay > 30) {
  return c.json({ error: 'Invalid lunar date' }, 400);
}
```
Some lunar months have 29 days (小月). Day 30 in those months is invalid but accepted; `getDeityDay` returns null for it (harmless). The validation contract is incorrect.

**Note:** Correct per-month day-count validation requires a lunar calendar lookup, which is disproportionate complexity for a read-only data route. Acceptable mitigation: tighten comment and add a test asserting null return for day 30.

---

### B5 · getMonthDays adjacent-month padding (leading cells mark wrong month) · MEDIUM
Same root as B1. `isCurrentMonth: false` would correctly mark them, but since the data comes from day 1 of the *current* month, the cells are also showing wrong dates. Tracked under B1.

## C — Robustness Gaps

### C1 · zodiacStorage: no animal validation · LOW
**File:** `src/utils/zodiacStorage.ts`
Any string can be stored as zodiac. On read-back in `auspiciousScan.ts`, an invalid animal never matches `clashAnimal`, so the clash filter silently passes (shows all dates as compatible). User might have corrupted storage from a dev build.

**Fix:** Validate against the 12 valid zodiac strings on read; return null for unrecognised values.

---

### C2 · scanChunk: silent empty result when maxDate < startDate · LOW
**File:** `src/utils/auspiciousScan.ts:17`
No guard. Callers pass sensible dates from UI, but a defensive check costs one line.

---

### C3 · buildShichen: luck via hash, not traditional rules · NOTE
**File:** `packages/shared/src/lunar/index.ts:buildShichen`
Shichen luck is computed via a hash of the date, not traditional stem-branch relationship rules. Results are deterministic per-date but may not match published almanac values. This is probably acceptable for the app's purpose but should be documented as an approximation.

---

### C4 · moonPhase: custom algorithm vs lunar-javascript built-in · NOTE
**File:** `packages/shared/src/lunar/index.ts:moonPhase`
Custom synodic calculation used for the moon phase display, but `lunar-javascript` exposes `getYueXiang()` (月相) which is already used in `DayData.tongshu.yueXiang`. The two could diverge for edge dates.
